# Plan Management & Progression

This document provides a comprehensive, code-level breakdown of how a user progresses through their "Legend" (learning plan). It explains the "Just-In-Time" data population strategy and the logic for module and plan completion.

---

## Part 1: Module Progression Logic

Progression is not automatic; it is validated at every step by the `check-module-completion` Edge Function.

### `check-module-completion` Edge Function
This function is called whenever a user completes a task in the UI.

**Logic Flow:**
1.  **Verification:** Validates the user's ownership of the plan.
2.  **Task Check:** Queries the `tasks` table for any row in the current `module_number` where `is_completed = false`.
3.  **Completion Detection:** If all tasks are completed:
    - **Plan Completion:** It checks if the `module_number` equals the `total_estimated_modules`. If so, it returns `PLAN_COMPLETED`.
    - **Subscription Gate:** If the plan is "complex," it verifies the user has an `active` or `pro` subscription before allowing them to continue.
    - **Trigger Next Module:** It checks if quests for the *next* module already exist in the `quests` table. If not, it invokes the `continue-plan` Edge Function to generate them.

---

## Part 2: "Just-In-Time" Generation (`continue-plan`)

The `continue-plan` function implements the core strategy of only populating the database as the user needs it.

### `continue-plan` Edge Function
**Logic Flow:**
1.  **Context Loading:** It reads the `plan_details` JSONB column from the `plans` table.
2.  **Targeting:** It finds the specific module object for the `module_to_generate`. It extracts the `theme` and `objectives` that were pre-generated by the Dean AI during the initial plan creation.
3.  **Progress Context:** It fetches a summary of the user's completed tasks to provide "memory" to the AI.
4.  **AI Invocation (`CONTINUE_PLAN_PROMPT`):**
    - Gemini is instructed to generate exactly 7 days of quests for the specific theme and objectives.
    - It enforces strict adherence to the learning objectives.
5.  **Data Population:** It `INSERT`s the new 7 `quests` and their associated 14-35 `tasks` into the database.
6.  **Narrative Backgrounding:** It triggers the `generate-quest-stories` function as a background process to create the fantasy flavor text for each of the new quests, maintaining the narrative thread.

---

## Part 3: Plan Completion (`check-plan-completion`)

When the final task of the final module is checked, the system recognizes the end of the journey.

### `check-plan-completion` Edge Function
**Logic Flow:**
1.  **Finality Check:** Verifies that the user is on the last module and that all tasks in the entire plan are `is_completed = true`.
2.  **Reward Granting:** If completed, it calls the `award_plan_completion_reward` PostgreSQL RPC.
3.  **RPC Logic:**
    - Updates `profiles.coins` with +500 Gold.
    - Creates a unique `shop_item` called the "Pathfinder's Trophy" (source: `reward`).
    - Adds the trophy to the user's `user_items` inventory.
    - Inserts a "Journey Complete!" notification.

---

## Part 4: Abandoning a Path (`abandon-active-plan`)

Users have the option to restart their journey entirely.

### `abandon-active-plan` Edge Function
This is a destructive function that performs a manual cascade delete of the user's entire game state.

**Logic Flow:**
1.  **Plan Wipe:** Deletes all `tasks`, `user_stats`, `quests`, and the `plans` record.
2.  **Profile Reset:** Sets `coins: 0`, `xp: 0`, and `level: 1` in the `profiles` table.
3.  **Inventory Wipe:** Clears `user_items` and `equipped_items`.
4.  **History Wipe:** Deletes all `chat_history` and `adventure_chat_history`.

This ensures that when a user starts a "New Path," they are doing so from a completely clean slate, preventing any carry-over from their previous attempt.
