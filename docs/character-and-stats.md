# Character & Stats

This document provides a comprehensive, code-level breakdown of the player progression system in Questr. It explains how experience is calculated, how level-ups are processed, and how the various types of stats function.

---

## Part 1: Player Levels and Global XP

Questr uses a centralized progression system stored in the `profiles` table.

### 1. The XP Curve
The amount of Experience Points (XP) required to reach the next level is calculated using a power-based formula:
- **Formula:** `100 * (CurrentLevel ^ 1.5)`
- **Implementation:** `get_xp_threshold(p_level)` (PostgreSQL Function)

| Level | XP to Next Level |
|:---|:---|
| 1 | 100 |
| 2 | 282 |
| 3 | 519 |
| 10 | 3,162 |

### 2. The Level-Up Process
Leveling is handled by the `process_level_up(p_user_id)` function.
- **Logic:** It compares the user's current XP against the threshold.
- **Iteration:** It uses a `LOOP` to handle multiple level-ups simultaneously if a large reward was granted.
- **Persistence:** It subtracts the threshold from the user's XP and increments their level, updating the `profiles` table.

## Part 2: The Reward Engine (`handle_task_rewards`)

Most progression is driven by the completion of daily tasks. This is automated via a database trigger on the `tasks` table.

### Logic Flow:
1.  **Completion Detection:** The trigger only fires when a task's `is_completed` column changes from `false` to `true`.
2.  **Context Fetching:** It identifies the user and the current module number of the plan.
3.  **Reward Calculation:**
    - **Gold:** The base `reward_coins` from the task.
    - **Action Points (AP):** Grants +5 AP (used for Dungeons).
    - **XP Formula:** `(Coins * 10) * (1 + ModuleNumber * 0.1)`. This ensures that tasks in later "Legend Chapters" provide significantly more experience than early ones.
4.  **Atomic Update:** It updates the `profiles` table with all rewards in a single transaction.
5.  **Hooks:**
    - It calls `process_level_up`.
    - It heals the user's pet (if applicable).
    - It triggers `check_achievements`.

## Part 3: Path-Specific Stats (`user_stats`)

Unlike the global Level, `user_stats` are generated by the AI Dean specifically for each user's goal.

### Purpose:
These stats (e.g., "Logic", "Focus", "Creative Flow") provide the context for the **AI Adventure DM**. When the DM requests a "Roll", it will typically target one of these plan-specific stats.

### Data Model: `user_stats` table
| Column | Type | Description |
|:---|:---|:---|
| `plan_id` | `bigint` | Links the stats to a specific learning journey. |
| `name` | `text` | The AI-generated name of the stat. |
| `value` | `integer` | The current value (starts at 10). |

## Part 4: Security & Atomicity

- **Authority:** All progression logic (rewards and levels) resides in the database as `SECURITY DEFINER` functions. This prevents "XP farming" or "Coin spoofing" from the client side.
- **Integrity:** Using a database trigger (`handle_task_rewards`) ensures that rewards are *never* missed when a task is completed, regardless of which UI component or function performed the update.
- **Snapshotting:** During Dungeon Generation, the app takes a `stats_snapshot` and `inventory_snapshot`. This ensures that the DM is using the correct, authoritative stats for the entire duration of the run.

## Part 5: The Achievement System

Questr features a multi-category badge system to recognize player milestones.

### 1. Catalog & Unlocks
- **`achievements`:** Master table defining every badge, its category (`Journey`, `Character`, `Economy`, `Engagement`), and its rewards (Gold/XP).
- **`user_achievements`:** Tracks which badges a user has earned.

### 2. The Recognition Engine (`check_achievements`)
This is a complex `SECURITY DEFINER` function that acts as a rules engine.
- **Triggering:** It is called after significant actions (e.g., completing a task, leveling up, buying an item).
- **Logic:** It calculates the user's total completed tasks, modules, and plans. It then iterates through all achievements the user *doesn't* have and checks their stats against the criteria.
- **Awarding:** If a match is found, it calls `award_achievement`, which inserts the unlock record and triggers a "success" notification.

## Part 6: The Leaderboard System

Questr uses a dedicated, secure function to rank players while protecting their privacy.

### `get_leaderboard(sort_by, limit_count)`
- **Security:** This function runs as `SECURITY DEFINER`. This is essential because standard users are blocked by RLS from reading other users' profiles. This function bypasses RLS but only returns a specific, safe subset of data (`full_name`, `xp`, `level`, `streak`, `character_model_path`).
- **Dynamic Sorting:** It supports sorting by either `xp` or `current_streak`.
- **Injection Protection:** It uses PostgreSQL's `format()` function with `%I` (identifier) and `%L` (literal) to safely construct the dynamic query, preventing SQL injection.
