# AI Asset Mapping & Utilities

This document provides a comprehensive, code-level breakdown of the "Magic Bridge" in Questr: the system that converts unstructured AI output into structured, visual game assets using weighted fuzzy matching and robust LLM orchestration.

---

## Part 1: LLM Orchestration (`_shared/llm.ts`)

The project uses a custom-built wrapper for the Google Gemini API to handle the specific needs of an interactive game.

### 1. Robust Standard Requests (`callGemini`)
- **JSON Enforcement:** It uses `response_mime_type: 'application/json'` to force the AI into returning structured data, minimizing parsing errors.
- **Immersion Over Sanity:** It explicitly disables all safety filters (`HARM_CATEGORY_...: OFF`). This is critical for generating immersive fantasy content involving violence or dark themes without triggering generic AI refusals.
- **Error Resilience:** It performs deep checks on the response structure (`candidates`, `parts`, `text`) before returning data, providing clear error messages if the AI returns an empty or malformed response.

### 2. The Streaming Engine (`streamGemini`)
Since LLMs can be slow, Questr uses a custom streaming implementation to provide instant feedback.
- **The Buffer Problem:** Standard HTTP streams return chunks of bytes, not clean JSON objects.
- **The Solution (JSON Balancer):** The utility implements a "naive JSON balancer" that monitors open and closed braces `{}`. It accumulates bytes into a buffer until it detects a complete, valid JSON object from the Gemini stream.
- **Extraction:** Once balanced, it parses the chunk, extracts the newly generated text, and `yields` it to the client, ensuring the user sees the Oracle's thoughts as they are being "forged."

## Part 2: The Asset Mapping Engine (`_shared/icon_utils.ts`)

This engine is responsible for assigning a high-quality icon to any item the AI creates (e.g., "Blade of Whispers").

### 1. The Weighted Search Algorithm (`findIconForItem`)
The mapper uses a sophisticated scoring system to find the best visual match:
- **Keywords:** It combines the item's name with specific keywords generated by a secondary LLM call (`KEYWORD_GENERATOR_PROMPT`).
- **Scoring Weights:**
    - **Filename Match (Premium):** +5 points (e.g., "blade" in the name).
    - **Tag Match (Premium):** +3 points (e.g., "blade" in the metadata tags).
    - **Name Keywords:** +2 or +1 points for secondary matches.
- **Selection:** The icon with the highest aggregate score is chosen.

### 2. Intelligent Placeholders
If the algorithm cannot find a strong match (score < threshold), it falls back to a pool of **121 abstract icons** (`abstract-001.png` through `abstract-121.png`). This ensures that even the most obscure AI-generated items still have a clean, thematic visual representation.

---

## Part 3: The Icon Knowledge Base (`_shared/icon_tags.ts`)

The mapping engine is powered by a massive metadata file containing over **4,000 assets**.
- **Structure:** A key-value map where the filename is the key and an array of synonyms/concepts is the value.
- **Example:**
  ```javascript
  "3d-hammer.png": ["3d", "hammer", "tool", "smith", "forge", "build", "craft"]
  ```
- **Scale:** This comprehensive tagging allows the engine to understand that a "Smithing Mallet" should probably use the `3d-hammer.png` asset, even if the words don't match exactly.

---

## Part 4: Architectural Invariants

- **Stateless AI:** The AI utilities remain stateless. All context (inventory, stats, history) must be injected into the prompt by the calling Edge Function.
- **Server-Side Rendering of Assets:** The mapping always occurs on the server (Edge Function) before the item is saved to the database. This ensures that the frontend receives a ready-to-render URL and doesn't have to perform expensive fuzzy matching on every page load.
- **Model Flexibility:** The utilities support passing a `model` parameter, allowing different features to use different models (e.g., `gemini-2.5-pro` for curriculum design vs. `gemini-2.5-flash-lite` for simple pet mission stories).
